* What

An MCP server that exposes the full Emacs =obarray= — every text-consuming
function — as MCP tools, introspected at runtime from a live daemon.

Two modes, one codebase:

| Mode | Tools | Purpose |
|------+-------+---------|
| =core= | ~50 | Practical daily use |
| =maximalist= | ~3600+ | Reductio ad absurdum |

The maximalist mode is the point: demonstrating by construction that naive
"enumerate all tools" MCP server design saturates any agent context window.
The critique only works if the thing works.

* Axiom

#+begin_quote
The MCP server does not know what Emacs can do. Emacs tells it.
#+end_quote

No function list is hardcoded. All tool definitions derive from runtime
=obarray= introspection via a live Emacs daemon. If you are hardcoding
function names anywhere except tests, you have violated the axiom.

* Prerequisites

- Emacs 28+ with daemon support (=emacs --daemon=)
- Python >= 3.11
- [[https://github.com/astral-sh/uv][uv]] (package manager)
- GNU Make (=gmake= on macOS)

* Quick Start

#+begin_src bash
# Clone
ghq get jwalsh/emacs-mcp-maximalist
# or: git clone https://github.com/jwalsh/emacs-mcp-maximalist

# Install Python dependencies
uv sync

# Start Emacs daemon (if not already running)
emacs --daemon

# Generate manifest from running daemon
gmake manifest

# Run the server (core mode)
gmake run-core

# Run tests (no daemon required)
gmake test

# Check environment health
gmake health
#+end_src

** Emacs with mcp.el

#+begin_src emacs-lisp

 "emacs-core" "uv"
 '("run" "--directory" "/path/to/emacs-mcp-maximalist"
   "python" "src/server.py" "functions-core.jsonl")
 :initial-callback (lambda (conn) (message "emacs-mcp connected"))
 :tools-callback (lambda (conn tools) (message "%d tools" (length tools))))
#+end_src

* Architecture

#+begin_example
Emacs daemon (obarray)
    ↓ introspect.el
functions-compact.jsonl (build artifact, JSONL)
    ↓ server.py loads
MCP Server (stdio)
    ↓ tools/call
escape.py → dispatch.py → emacsclient --eval
    ↓
result → MCP content block
#+end_example

| Layer | File | Role |
|-------+-------------------+--------------------------------------|
| L0 | =emacs --daemon= | Running Emacs (prerequisite) |
| L1 | =src/introspect.el= | Walk obarray, emit manifest |
| L2 | =functions-compact.jsonl= | Build artifact (JSONL, not committed) |
| L3 | =src/escape.py= | Safe Elisp string literal builder |
| L4 | =src/dispatch.py= | emacsclient subprocess boundary |
| L5 | =src/server.py= | MCP server over stdio |
| L6 | =bin/health-check.sh= | Structured health check |

** Security Boundary

The escaper (=src/escape.py=) is the security surface. It handles
backslash escaping, quote escaping, null byte rejection, and shell
metacharacter safety. It is tested independently of the server.

** Manifest Format Invariant

| Format | File | Structure | Producer | Consumer |
|--------+-------------------------+--------------------------------------+----------------------------+------------|
| Full JSON | =emacs-functions.json= | ={"functions": [...]}= | =emcp-write-manifest= | debugging |
| Compact JSONL | =functions-compact.jsonl= | one ={"n","s","d"}= per line | =emcp-write-manifest-compact= | server.py |
| Core JSONL | =functions-core.jsonl= | filtered subset, same format | =emcp-write-manifest-core= | server.py |

* Build Pipeline

The project uses a sentinel-gated Makefile pipeline for bootstrapping:

#+begin_src bash
gmake status        # show pipeline + artifact status
gmake graph         # show dependency DAG
gmake work          # run full pipeline (bootstrap → implement)
gmake -j3 parallel  # run independent phases concurrently
gmake test          # run all tests (53 cases)
gmake manifest      # generate both manifests from emacs daemon
gmake run-core      # start MCP server (core mode)
gmake run-max       # start MCP server (maximalist mode)
gmake health        # run environment health check
gmake sync          # export README.org → README.md
#+end_src
