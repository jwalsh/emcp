## What the Meta-Meta Prompt Produced

The gist contains a 4-level meta-tower for bootstrapping repos:

| Level | Document | Purpose |
|-------|----------|---------|
| 0 | Code | The implementation |
| 1 | CLAUDE.md | Agent instructions for one repo |
| 2 | meta-meta-prompt.md | Template for generating CLAUDE.md |
| 3 | meta-meta-meta-prompt.md | Protocol for new domain classes |
| 4 | meta⁴ prompt | When to stop going meta |

This session used level 2 (the meta-meta prompt) to generate level 1
(CLAUDE.md) from level 0's specification (spec.org). The process:

1. Read spec.org, extract axiom/anti-goals/build-order/conjectures
2. Generate CLAUDE.md with 4 kernel constraints (axiom before line 10,
   confirmation gate, failure handler, instrumentation requirement)
3. Self-review against critical/substantive/minor checklist
4. Wire the backlog (issues + conjecture tracking)
5. Create persistent memory files
6. Verify with 5-question test
7. Decompose into parallel work streams
8. Implement

## Skills as Pipeline Phases

Eight Claude Code skills in `.claude/skills/` map to the meta-meta
prompt's 11 phases. Each skill has a sentinel guard (idempotency check)
and is invoked non-interactively via `claude --dangerously-skip-permissions -p`.

The Makefile wires them into a DAG with non-phony sentinel files:

```
spec.org → bootstrap → generate-claude-md → review-prompt
  → {wire-backlog, setup-memory, health-check}  (parallel)
  → verify-bootstrap → decompose → work
```

`gmake work` runs the full pipeline. `gmake -j3 parallel` exploits
the three independent post-review phases. `gmake force-<phase>` re-runs
a single phase by deleting its sentinel.

## Corrections Applied During Session

The session produced several real-time corrections — evidence that the
protocol surfaces problems rather than hiding them:

1. **JSON/JSONL format drift** — spec.org's `emcp-write-manifest`
   produces JSON, but the server expected JSONL. Caught during CLAUDE.md
   generation. Fix: added manifest format invariant table and
   `emcp-write-manifest-compact` function.

2. **Git notes convention** — initial attempt blindly copied a version
   string. User corrected: adopt the existing aq project convention
   (X-Agent-Role, X-Agent-Runner, etc.). Research agent confirmed the
   convention was already operational in other repos.

3. **Co-author in trailers vs notes** — initial commits used
   `--trailer "Co-authored-by: ..."`. User corrected: co-author
   attribution belongs in git notes, not commit messages. Clean
   messages, metadata out-of-band.

4. **Primary user not stated** — verify-bootstrap found 4/5 questions
   answerable from CLAUDE.md. Gap: "who is the primary user" required
   reading spec.org. Fix: added explicit statement to CLAUDE.md.

5. **Idempotency** — initial skills had no sentinel checks. User asked
   "are those idempotent / have sentinels?" Answer: no. Fixed all eight
   with precondition checks.

## What This Demonstrates

The meta-meta prompt protocol is not a planning document. It is an
executable specification for agent bootstrapping. Given a `spec.org`
and the protocol, a single session can produce:

- A complete project scaffold with agent-readable instructions
- An issue backlog with dependency ordering
- Falsifiable conjectures with measurement hooks
- A working implementation with tests
- Persistent memory for session continuity

The protocol's value is in its corrections: format invariants caught
before they became runtime bugs, idempotency gaps filled before they
caused duplicate work, and verification questions that surface gaps in
the agent's understanding.

The four kernel constraints (axiom placement, confirmation gate, failure
handler, instrumentation requirement) survived all 22 commits. They are
the fixed point.
