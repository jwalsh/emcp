* Summary

Four new specs extend emacs-mcp-maximalist with daemon-as-retrieval-layer
(JITIR), eval trust boundaries, agent presence/discovery via mDNS, and
OTel observability.  Together they introduce:

- *34 new conjectures* (C-D001--C-D008, C-E001--C-E010, C-F001--C-F010,
  C-G001--C-G006) extending the existing C-001 through C-011
- *~20 implementation tasks* across Elisp, Python, infrastructure, and docs
- *2 axiom conflicts* that require architectural decisions before proceeding

All items are registered as beads in the project tracker (~bd list~).

* Axiom Compliance Review

** CLAUDE.md Axiom: "The MCP server does not know what Emacs can do. Emacs tells it."

The new specs are generally compliant.  The query library, eval barrier,
presence tools, and metrics module all live in Elisp and are loaded into
the Emacs process.  No component outside Emacs defines or augments the
function list.

** Conflict 1: =src/emcp_otel.py= — Python in the Observability Stack

The =emcp-observability.org= spec introduces =src/emcp_otel.py=, a Python
process that scrapes daemon state via =emacsclient= and exports OTel
metrics.  This contradicts the "Python-first architecture" anti-goal in
CLAUDE.md:

#+begin_quote
Emacs Lisp owns introspection and dispatch.  Python is the MCP protocol
shim only.  If logic migrates from Elisp to Python, the axiom is being
violated.
#+end_quote

*Assessment*: The OTel bridge is NOT in the MCP server path.  It is a
sidecar that reads metrics via =emacsclient=, not a protocol shim.  The
axiom specifically targets logic migration from Elisp to Python.
Observability scraping is not introspection or dispatch logic.

*Recommendation*: Accept Python for =emcp_otel.py= with the following
scoping constraint: the bridge may only READ state (via =emacsclient --eval=),
never WRITE or DISPATCH.  Document this as a deliberate exception in
CLAUDE.md's stack preferences table.

*Alternative*: Pure Elisp metrics export via =emcp-metrics/snapshot=
(already spec'd) returning JSON that a generic OTel exporter can scrape.
This keeps Elisp as the sole authority but requires an HTTP endpoint or
file-based export.

Bead: =emacs-mcp-maximalist-ek2=

** Conflict 2: =emcp-presence/query-peer= uses =shell-command-to-string=

The =emcp-presence.el= spec's =query-peer= function calls
=shell-command-to-string= to SSH into a remote daemon.  However,
=emcp-eval-barrier.el= blocks =shell-command-to-string= at Tier 2+.
This is a self-referential conflict: the presence layer needs the
exact capability the eval barrier forbids.

*Recommendation*: =query-peer= should use =call-process= (which can be
individually allowlisted in the barrier as an internal-only exception)
or route through =emcp-t2/remote-eval= which uses =shell-command-to-string=
internally but is a named library function, not arbitrary eval.

Bead: =emacs-mcp-maximalist-pem=

** Conflict 3: Spec tangle targets create files that are build artifacts

Multiple specs use =:tangle yes= which would create Elisp files directly
in =src/=.  Per CLAUDE.md, the manifest is a build artifact not committed.
Similarly, these tangled files should be treated as generated output from
the org specs, not committed source.  The implementation should either
tangle-and-commit (making the org file the source of truth) or write the
Elisp manually (making the org file documentation).

*Recommendation*: Tangle-and-commit.  The org spec is the source of
truth.  The tangled =.el= files are the implementation.  This is standard
literate programming practice.

* Architecture Decision Record: Python in Observability

** Context

=emcp-observability.org= specifies =src/emcp_otel.py=, a Python sidecar
that scrapes Emacs daemon state and exports OTel metrics.  The parent
project (CLAUDE.md) has an explicit anti-goal: "Python-first architecture."

** Decision Needed

Should the OTel bridge be:

A. *Python (as spec'd)* — mature OTel SDK, natural subprocess/HTTP model,
   but introduces a Python dependency outside the MCP server path.

B. *Pure Elisp* — =emcp-metrics/snapshot= writes JSON to a file or HTTP
   endpoint; a generic OTel collector scrapes it.  No Python needed.
   Cost: must implement HTTP server in Elisp or use file-based export
   (polling from collector).

C. *Hybrid* — Elisp generates the metrics snapshot, a 20-line shell
   script curls it to the OTel collector.  No Python, no Elisp HTTP.

** Recommendation

Option A (Python) with the scoping constraint that =emcp_otel.py= is
marked as a sidecar, never loaded by =emcp-stdio.el=, and may only READ
daemon state.  This is consistent with the axiom: Python is acceptable
for infrastructure tooling that does not participate in MCP dispatch.

** Dissent path

If the team prefers pure Elisp: Option C (shell bridge) is the lowest
cost alternative.  =emcp-metrics/snapshot= already produces the JSON;
a cron job with =curl= pushes it.

* Sequencing Recommendation

1. Implement =emcp-eval-barrier.el= first (P0, security surface)
2. Implement =emcp-query-library.el= (P0, highest user value)
3. Wire barrier into daemon eval dispatch (requires understanding
   =emcp-stdio.el= dispatch without modifying it — use advice or
   a wrapper)
4. Implement =emcp-metrics.el= (P1, low risk)
5. Implement =emcp-presence.el= (P1, depends on barrier)
6. Deploy Avahi service file on nexus (P2, enables P3 validation)
7. Deploy OTel bridge + Grafana stack (P2)
8. Begin homelab multi-host testing for P3 conjectures

Total estimated effort: P0 is 2-3 sessions.  P1 is 2-3 sessions.
P2 is 1-2 sessions.  P3 is open-ended research.
