* Emacs All The Way Down: An Experience Report

An MCP server that exposes the full Emacs =obarray= as tools.

- Started: Python shim wrapping =emacsclient=
- Ended: Pure Elisp MCP server --- ~280 lines, no external deps
- Duration: One session (2026-03-17 to 2026-03-18)
- 27+ agents, 36 commits, 11 conjectures

* The Axiom

#+begin_quote
The MCP server does not know what Emacs can do. Emacs tells it.
#+end_quote

** All tool definitions derived at runtime from =obarray= introspection
** No function list is hardcoded
** If you are hardcoding function names, you have violated the axiom

Five layers between the MCP client and Emacs:

| Layer | File | Role |

** =escape.py= was the security surface --- 11 edge cases, independently tested

** Two runtime modes: core (~50 tools) and maximalist (~3600+ tools)

* The Pivot

"What if Emacs IS the server?"

** Unicode round-trip test (C-004) sparked the idea
** 14 character classes: CJK, emoji, ZWJ families, Arabic, combining chars
** All byte-identical through =prin1-to-string= / =funcall= / stdout
** If Emacs can serialize, dispatch, and respond --- why have Python at all?

* emcp-stdio.el

~280 lines of Emacs Lisp. One file. No dependencies.

** Reads JSON-RPC from stdin via =json-parse-string=
** Walks =obarray= at startup --- builds tool defs for text-consumer functions
** =funcall= dispatches directly --- no subprocess, no manifest
** =send-string-to-terminal= writes responses to stdout (unbuffered)
** Detects running daemon --- adds 9 data-layer tools via =emacsclient=

* C-011: Stdout Buffering --- The Real Bug

=princ= + =terpri= in Emacs batch mode does not flush stdout to a pipe.

** 4 agents investigated, 6 approaches tested

| Approach | Works? | Notes |
|------------------------------------+--------+--------------------------------------|
| =princ= + =terpri= | NO | libc stdio fully buffered on pipes |
| =princ= + =sit-for 0= | NO | sit-for does not trigger fflush |
| =send-string-to-terminal= | YES | Direct fd 1 write, cross-platform |
| =write-region /dev/stdout= | YES | Bypasses libc via write(2), Unix only |
| =write-region /dev/fd/1= | YES | Same mechanism, macOS/Linux |
| =call-process-region= + =cat= | NO | Output goes to standard-output var |

** =send-string-to-terminal= chosen --- writes directly to fd 1
** Secondary fix: daemon check hanging on unreachable remote sockets

* C-006 Deep Dive

Vanilla (-Q) = 8460 functions. Configured = 10040. Delta = 1580 (18.7%).

** 5 hypotheses, all confirmed

| Finding | Detail |
|-------------------------------------+-----------------------------------------|
| =--batch= implies =-q= | No user init loaded in batch mode |
| Filter masks 93% of delta | Heuristic reduces 1580 to ~110 extra |
| 222 hidden autoloads | =help-function-arglist= returns sentinel |
| C primitives identical (1336) | init.el adds only Elisp-defined fns |
| Category error in test | Vanilla vs configured test was comparing wrong scope |

** The text-consumer heuristic acts as a natural stabilizer
** Core tool set is stable regardless of user configuration
** 780 tools = ~9.2% of vanilla functions, ~7.8% of configured

* C-009: The Autoload Barrier

222 text-consumer functions hidden by unavailable arglists.

** Problem: =help-function-arglist= returns sentinel for unloaded autoloads
** Resolving would yield 28.8% increase in tool count

5 approaches under investigation:

| Approach | Mechanism | Risk |
|-------------------------------+------------------------------+------------------------|
| A: Selective autoload-do-load | Force-load matching symbols | Side effects on load |
| B: Parse autoload objects | Extract arglist from plist | Rarely present |
| C: Parse .elc headers | Read bytecode without eval | Fragile, circular dep |
| D: Bulk-load loaddefs.el | Replay normal startup | Violates -Q constraint |
| E: Static arglist map | Grep .el source at build time | Stale if Emacs updates |

* What We Proved

** Emacs can serve MCP natively --- no Python, no Node, no subprocess shim
** The =obarray= IS the tool registry --- no manifest file needed
** =funcall= IS the dispatch --- no s-expression builder needed
** =prin1-to-string= IS the security boundary --- no custom escaper needed
** ~280 lines of Elisp replaces ~2381 lines of Python/bash
** The reductio works: 779 tools exposed, agent context not saturated (C-001)
** CPRR (Conjecture-Prediction-Refutation-Refinement) drives implementation

* What We Discovered

** Batch mode stdout buffering (C-011)
*** =princ= does not flush to pipes --- protocol-breaking for interactive MCP
*** =send-string-to-terminal= is the cross-platform fix

** Daemon instability under concurrent agents
*** 3 daemon PIDs in 2 minutes with 9 agents
*** No coordination protocol for daemon lifecycle

** Text property leaks
*** =json-serialize= returns unibyte strings in batch mode
*** Requires =decode-coding-string= before output

** Arglist heuristic is imprecise (C-002)
*** 40% precision vs. 80% conjectured
*** =object=/=buffer= args dominate false positives

* Open Questions

** C-009: Autoload barrier
*** 222 functions hidden --- 5 approaches, none clearly safe
*** Selective loading risks side effects; static parsing is fragile

** Daemon coordination
*** Multiple agents spawn competing daemons
*** Need lock file or shared daemon protocol
*** The =aq= gossip layer is a potential solution

** =org-element-parse-buffer= as structured data
*** Org files are semi-structured --- parse tree gives agents typed access
*** Currently returning flat text from daemon tools

** Arglist heuristic refinement
*** 40% precision is not good enough (C-002 refuted)
*** Tighter patterns? Exclude =object=/=buffer= from predicates?
*** Or: accept the false positives --- the reductio benefits from more tools
