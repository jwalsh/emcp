* Overview

=emcp-stdio.el= is a pure Emacs Lisp implementation of an MCP (Model Context Protocol) server.
It reads newline-delimited JSON-RPC 2.0 messages from stdin and writes responses to stdout,
running Emacs in =--batch= mode.  Diagnostics go to stderr via =message=.

The file eliminates the Python stack (layers L1--L5 of the original architecture) by making
Emacs simultaneously the MCP protocol handler, the function authority, and the dispatch engine.
See [[file:conjectures/c-008-research.md][C-008]] for the conjecture and measurements behind this collapse.

** Invocation

#+begin_src shell
# Core mode: vanilla Emacs, no user init, ~779 tools
emacs --batch -Q -l src/emcp-stdio.el -f emcp-stdio-start

# Maximalist mode: user init loaded, ~3600+ tools
emacs --batch    -l src/emcp-stdio.el -f emcp-stdio-start
#+end_src

The =-Q= flag controls which =obarray= symbols are available.  With =-Q=, only built-in
and autoloaded functions are present.  Without it, =init.el= packages add to the obarray,
increasing tool count.  See [[file:../CONJECTURES.md][CONJECTURES.md]] C-006 for the vanilla vs. configured measurement.

** Runtime Modes

| Mode       | Flag  | Tool count | Use case                          |
|------------+-------+------------+-----------------------------------|
| Core       | =-Q=  | ~779       | Reproducible, portable, fast      |
| Maximalist | (none)| ~3600+     | The reductio ad absurdum demo     |
| +Daemon    | any   | +9         | Data layer tools when daemon runs |

Mode is determined by Emacs startup flags, not by code branches.  Daemon tools are
additive: if =emacsclient= can reach a running daemon, 9 data layer tools are appended
to the tool set regardless of mode.

| Python layer        | Replaced by                  | How                                      |
|---------------------+------------------------------+------------------------------------------|
| =src/server.py= (L5)   | [[file:../src/emcp-stdio.el::defun emcp-stdio-start][emcp-stdio-start]]             | JSON-RPC read loop in batch mode         |

| =src/dispatch.py= (L4) | =funcall= / [[file:../src/emcp-stdio.el::defun emcp-stdio--daemon-eval][emcp-stdio--daemon-eval]] | In-process for local; emacsclient for daemon |

* Architecture

#+begin_example
                         +--------------------------+
                         |  MCP Client (Claude Code)|
                         +------------+-------------+
                                      | stdin (JSON-RPC)
                                      v
                    +----------------------------------+
                    |  emacs --batch -Q -l emcp-stdio  |
                    |                                  |
                    |  emcp-stdio-start                |
                    |    |                             |
                    |    +-- set UTF-8 encoding        |
                    |    +-- check daemon              |
                    |    +-- collect-tools (mapatoms)  |
                    |    +-- build daemon tools        |
                    |    +-- read loop:                |
                    |         |                        |
                    |         +-- read-line (stdin)    |
                    |         +-- json-parse-string    |
                    |         +-- dispatch             |
                    |              |                   |
                    |              +-- initialize      |
                    |              +-- ping            |
                    |              +-- tools/list      |
                    |              +-- tools/call      |
                    |                   |              |
                    |            +------+------+       |
                    |            |             |       |
                    |         local          daemon    |
                    |        (funcall)    (emacsclient)|
                    |            |             |       |
                    |            v             v       |
                    |         result        result     |
                    |            \            /        |
                    |             +-- respond          |
                    |                   |              |
                    +-------------------+--------------+
                                        | stdout (JSON-RPC)
                                        v
                         +--------------------------+
                         |  MCP Client              |
                         +--------------------------+
#+end_example

* Data Flow

#+begin_example
stdin
  |
  v
read-line  -----> raw string (one JSON line)
  |
  v
json-parse-string  -----> alist (id, method, params)
  |
  v
dispatch
  |
  +--[initialize]--> respond(id, serverInfo+capabilities)
  |
  +--[ping]--------> respond(id, {})
  |
  +--[tools/list]--> respond(id, tools-cache vector)
  |
  +--[tools/call]--> route by name prefix:
  |     |
  |     +--[emcp-data-*]--> daemon-call --> daemon-eval --> emacsclient
  |     |                                                      |
  |     +--[other]---------> intern-soft --> eval(read(sexp)) -+
  |                                                            |
  |                                                     result string
  |                                                            |
  +-----------------------------------------------------------+
  |
  v
respond(id, content[{type:"text", text:result}])
  |
  v
send  -----> json-serialize --> decode-coding-string --> princ --> stdout
#+end_example

:PROPERTIES:
:CUSTOM_ID: handle-tools-call
:END:

- Source :: [[file:../src/emcp-stdio.el::defun emcp-stdio--handle-tools-call][emcp-stdio--handle-tools-call]]
- Arguments ::
  - =ID= -- JSON-RPC request ID
  - =PARAMS= -- Alist with =name= (tool name string) and =arguments= (alist with =args= vector)
- Returns :: =nil= (responds via [[file:../src/emcp-stdio.el::defun emcp-stdio--respond][emcp-stdio--respond]])
- Side effects :: Evaluates Elisp code (local tools) or spawns =emacsclient= (daemon tools); writes response to stdout
- Dispatch logic ::
  1. If =name= starts with ="emcp-data-"=: routes to [[file:../src/emcp-stdio.el::defun emcp-stdio--daemon-call][emcp-stdio--daemon-call]] (requires daemon)
  2. Otherwise: constructs s-expression string from name and args, evaluates via =eval(read(...))=
- Error handling :: All errors are caught and returned as MCP content with ="error: ..."= text (not as JSON-RPC errors), keeping the response within MCP tool call semantics.
- Security note :: Local tool dispatch uses =eval(read(sexp-str))= where =sexp-str= is built from the tool name (an interned symbol) and =%S=-formatted args.  The =%S= format provides basic quoting but this is not a hardened security boundary.

:PROPERTIES:
:CUSTOM_ID: start
:END:

- Source :: [[file:../src/emcp-stdio.el::defun emcp-stdio-start][emcp-stdio-start]]
- Arguments :: None
- Returns :: Does not return until stdin closes (EOF)
- Side effects ::
  1. Sets language environment to UTF-8
  2. Sets =coding-system-for-read= and =coding-system-for-write= to =utf-8=
  3. Configures binary mode on stdin/stdout (if =set-binary-mode= is available)
  4. Probes for running daemon via [[file:../src/emcp-stdio.el::defun emcp-stdio--check-daemon][emcp-stdio--check-daemon]]

Parse errors and dispatch errors are logged to stderr but do not terminate the loop.

*** emcp-data-eval

The raw query interface.  Evaluates arbitrary Elisp in the running daemon.  This is the most powerful and least safe daemon tool.

#+begin_src elisp
;; sexp template (passed directly to emacsclient --eval):
(car args)  ; the sexp-string argument, evaluated as-is
#+end_src

*** emcp-data-buffer-list

Lists all open buffers with their associated filenames.

#+begin_src elisp
;; sexp template:
(mapconcat (lambda (b)
             (format "%s\t%s" (buffer-name b)
                     (or (buffer-file-name b) "")))
           (buffer-list) "\n")
#+end_src

*** emcp-data-org-headings

Extracts all org headings with level, TODO state, title, and tags.

(progn (require 'org)

    (mapconcat (lambda (c) (mapconcat #'identity c "\t"))
               (org-map-entries
                (lambda () (let ((h (org-heading-components)))
                             (list (make-string (nth 0 h) ?*)

Appends a new org entry under a heading, or creates a top-level heading if not found.

  (with-current-buffer (find-file-noselect FILE)

        (progn (org-end-of-subtree t)
               (insert "\n** " BODY "\n")

             (insert "\n* " HEADING "\n" BODY "\n")

* Known Gotchas
:PROPERTIES:
:CUSTOM_ID: gotchas
:END:

** json-serialize unibyte output
:PROPERTIES:
:CUSTOM_ID: gotcha-unibyte
:END:

=json-serialize= returns a *unibyte* string containing raw UTF-8 bytes.  In batch mode,
=princ= on a unibyte string may interpret the bytes as Latin-1, producing mojibake for
any non-ASCII content.

The fix in [[file:../src/emcp-stdio.el::defun emcp-stdio--send][emcp-stdio--send]]:

#+begin_src elisp
(princ (decode-coding-string (json-serialize alist) 'utf-8))
#+end_src

This converts the raw UTF-8 bytes into a proper multibyte Emacs string before output.
See [[file:reference/parsing-json.md][parsing-json reference]] and [[file:reference/coding-system-basics.md][coding-system-basics reference]].

** emacsclient diagnostic filtering
:PROPERTIES:
:CUSTOM_ID: gotcha-emacsclient
:END:

When the daemon is reached via TCP (rather than Unix socket), =emacsclient= may emit
diagnostic lines like =Waiting for Emacs...= to its stdout.  These lines are mixed
into the eval result.

[[file:../src/emcp-stdio.el::defun emcp-stdio--daemon-eval][emcp-stdio--daemon-eval]] filters any line containing ="emacsclient:"= before
returning the result.  This mirrors the same filtering in =src/dispatch.py=.

** Batch mode encoding setup
:PROPERTIES:
:CUSTOM_ID: gotcha-encoding
:END:

Emacs batch mode does not automatically use UTF-8 for stdio.  [[file:../src/emcp-stdio.el::defun emcp-stdio-start][emcp-stdio-start]]
explicitly configures:

1. =set-language-environment "UTF-8"= -- sets default coding systems
2. =coding-system-for-read= / =coding-system-for-write= to =utf-8=
3. =set-binary-mode= on stdin/stdout set to =nil= (text mode, not raw binary)

Without step 3, some platforms may use raw binary mode which bypasses encoding
entirely.  See [[file:reference/batch-mode.md][batch-mode reference]].

** read-from-minibuffer in batch mode
:PROPERTIES:
:CUSTOM_ID: gotcha-minibuffer
:END:

[[file:../src/emcp-stdio.el::defun emcp-stdio--read-line][emcp-stdio--read-line]] uses =read-from-minibuffer= to read stdin.  In batch mode,
this function reads a raw line from stdin (no prompt, no minibuffer display).
The empty string prompt =""= is required but not displayed.

** eval(read(...)) for local tool dispatch
:PROPERTIES:
:CUSTOM_ID: gotcha-eval-read
:END:

[[file:../src/emcp-stdio.el::defun emcp-stdio--handle-tools-call][emcp-stdio--handle-tools-call]] constructs an s-expression string and evaluates it
via =eval(read(...))= rather than using =funcall= directly.  This means arguments
arrive as strings (from JSON) and are interpolated with =%S= formatting.  While =%S=
provides Elisp-level quoting, this path is less safe than direct =funcall= with
properly typed arguments.  For daemon tools, =prin1-to-string= provides equivalent
quoting in [[file:../src/emcp-stdio.el::defun emcp-stdio--daemon-build-sexp][emcp-stdio--daemon-build-sexp]].

** Notification handling
:PROPERTIES:
:CUSTOM_ID: gotcha-notifications
:END:

JSON-RPC notifications have no =id= field.  [[file:../src/emcp-stdio.el::defun emcp-stdio--dispatch][emcp-stdio--dispatch]] swallows them
silently.  This correctly handles =notifications/initialized= but means any
malformed request missing an =id= is silently dropped.

* Invariants
:PROPERTIES:
:CUSTOM_ID: invariants
:END:

The following invariants must hold for correct operation:

1. *JSON-RPC 2.0 framing* -- Each message is a single line of valid JSON.  No embedded newlines within a message.  Messages are delimited by =\n=.

2. *UTF-8 encoding* -- All stdin/stdout data is UTF-8 encoded.  The server sets this explicitly at startup.  Non-UTF-8 input will cause parse errors.

3. *One-line-per-message* -- [[file:../src/emcp-stdio.el::defun emcp-stdio--read-line][emcp-stdio--read-line]] reads exactly one line.  [[file:../src/emcp-stdio.el::defun emcp-stdio--send][emcp-stdio--send]] writes exactly one line (via =terpri=).  No Content-Length headers are used (unlike LSP).

4. *Tools cache immutability* -- [[file:../src/emcp-stdio.el::defvar emcp-stdio--tools-cache][emcp-stdio--tools-cache]] is built once at startup and never modified.  =tools/list= returns the same result for the lifetime of the server process.

5. *Symbol interning* -- Local tool dispatch requires the tool name to resolve via =intern-soft= to an =fboundp= symbol.  If the obarray is modified after startup (unlikely in batch mode), a tool listed in =tools/list= could become un-callable.

6. *Daemon stability* -- Daemon tools assume the daemon remains reachable after the startup check.  If the daemon exits mid-session, daemon tool calls will fail with =emacsclient= errors.

7. *Argument type* -- All tool arguments arrive as JSON strings (=type: "array"= of =type: "string"=).  Functions expecting non-string arguments (integers, symbols) receive string representations.  The =eval(read(...))= path in local dispatch handles some coercion via the Elisp reader, but daemon tool sexp builders assume string arguments.

8. *No concurrent requests* -- The read loop is single-threaded and sequential.  Each message is fully processed before the next is read.  No request pipelining or out-of-order responses.

9. *Stderr for diagnostics only* -- =message= output goes to stderr in batch mode.  Stdout is exclusively for JSON-RPC responses.  Mixing diagnostic output into stdout would corrupt the protocol stream.

10. *Server internals excluded* -- [[file:../src/emcp-stdio.el::defun emcp-stdio--collect-tools][emcp-stdio--collect-tools]] explicitly filters out symbols starting with ="emcp-stdio-"=.  This prevents the server from exposing itself as a tool.
